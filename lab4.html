Lab 4: Interrupt-Driven Input and Timed Facilities

Turn a button edge into a debounced notification, then make the Overbug round and workstations progress on independent timers.

Before you begin

Linux / Raspberry Pi OS only Mission 1 uses a Linux kernel module, GPIO IRQs, and SIGIO. Mission 2 requires a repeating timing source that wakes the server. Windows support is not required.

Before starting this lab, make sure you have:

  • completed Lab 2's GPIO wiring and tested its reader on each Raspberry Pi controller;
  • a completed Lab 3 server for two to four players, ready to copy into Lab4/server.c; and
  • the clean Linux 6.1 source tree used to cross-compile the Lab 4 kernel module.

This lab assumes you understand:

  • the Lab 2 GPIO driver, reader, and OBIN packet path;
  • the Lab 3 server's shared game state and OBST display updates;
  • basic ideas of interrupts, timers, and UDP communication;

By the end, you will replace input polling with interrupt-driven input, debounce physical buttons, and use a 50 ms server timer to update held movement and the remaining time of each active facility.

1. Mission 1 — replace polling with IRQ, timer, and signal

The device name and text protocol do not change: /dev/overbug_buttons still returns OBTN XX\n. What changes is when the reader receives it. Remove the Lab 2 --hz option and all fixed-rate sleep logic. 

GPIO edge
  -> IRQ handler: wake the shared debounce timer
  -> timer callback: read six GPIO values and compare stable mask
  -> changed mask: store state, wake wait queue, signal SIGIO to reader
  -> reader SIGIO handler: wake the event loop
  -> reader read(): OBTN XX, then send one OBIN UDP packet

Driver contract

ComponentRequired behavior
GPIO setupRequest all six lines with gpio_request() and configure them with gpio_direction_input().
IRQMap each line with gpio_to_irq(), then request both rising and falling edges. The handler must not read GPIO or notify userspace; it only restarts the shared debounce timer.
Debounce timerUse one timer_list and the read-only module parameter debounce_ms, defaulting to 20. The callback reads the complete mask with gpio_get_value() and publishes it only if it differs from the last delivered mask.
Device readA normal read() blocks until a changed debounced mask is ready.
NotificationImplement a wait queue. Store the new state first, then call wake_up_interruptible() and kill_fasync(..., SIGIO, POLL_IN).
CleanupOn unload: prevent new opens, deregister the device, disable and free every IRQ first, stop the timer with del_timer_sync(), and call gpio_free() for every requested line.

Reader contract

The reader is an event bridge, not a polling loop. SIGIO is a doorbell that means “the device may have a new state”; it does not contain the button mask. The reader obtains the authoritative state with read(), converts OBTN XX\n into the six-byte OBIN packet, and sends it to the game server. 

driver stores a changed stable mask
  -> driver sends SIGIO                 (doorbell only)
  -> reader handler sets a flag         (no read or UDP here)
  -> reader event loop reads OBTN XX    (real button data)
  -> reader sends OBIN to the server

Build and exercise Mission 1

Cross-compile the kernel module in the Ubuntu VM against the clean 6.1 source tree. The Pi installation does not include a /lib/modules/$(uname -r)/build tree. 

# Ubuntu VM
cd Lab4
make KDIR=/path/to/linux-rpi-6.1-gpio \
    ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- module
scp overbug_buttons.ko <pi-user>@<pi-address>:~/Lab4/
# Raspberry Pi
cd ~/Lab4
make reader
sudo insmod overbug_buttons.ko debounce_ms=20
./reader --server <PC_IP> --player 1

A held direction remains active at the server until the release edge sends a new mask. That is why the reader no longer needs a 60 Hz polling loop.

2. Mission 2 — timed multiplayer facilities

Copy your final Lab 3 server into Lab4/ first. Keep its multiplayer behavior and synchronization design, then add the timing rules below. 

cd Lab4
cp /path/to/your/completed/Lab3/server.c server.c
make server

Required timing design

The game must take one small step every 50 ms. Make a repeating timer that tells the server when a step is due. The timer's job is only to wake the server. Do all game work in the normal server code: move players, update machines, and send the latest game state there.

Use each 50 ms game step to calculate the remaining time for the round and for every active welder and verifier. Each facility needs its own remaining-time value, because several facilities may be working at the same time. A welder counts down only while a player is soldering; a verifier counts down by itself.

Sometimes the server will be busy and miss one or more timer notices. Do not lose that time. Each time the server wakes up, use a clock that only moves forward to find how many 50 ms steps have really passed. Run all of those steps before sending the next game update. Keep all countdowns as whole milliseconds. 

Every 50 ms: tell the main server that a game step is due.

When the main server runs:
    work out how many 50 ms steps have passed
    reduce the remaining time of every active facility for every missed step
    run the matching game updates
    send the current game state

Continuous movement from event-driven input

A held direction button must keep a player moving, but the reader must not send repeated UDP packets while it is held. The driver sends one event when the button is pressed and another when it is released. When the server receives an OBIN packet, save that player's current button state. Every 50 ms, move each player using the saved state. When the player releases the direction button, the next 50 ms step stops the player.

The server keeps listening for OBIN packets, even between 50 ms game steps. When a packet arrives, it immediately replaces that player's saved button state. It does not move the player at that moment. When the next game step is due, the server uses the newest saved state to update the game. 

reader:  press Right -> OBIN(Right) ---------------- release -> OBIN(no Right)
server:                save Right mask                         clear Right mask
50 ms tick:            move -> move -> move -> move -> stop

In other words, do not move a player as soon as a packet arrives. A packet only records which buttons are currently held. The 50 ms game step moves players and updates the machines. This makes held movement smooth without making the reader send packets again and again.

Timed objectStart and pause ruleCompletion
RoundStart at 180,000 ms (3 minutes). Count down on every 50 ms step.Stop at zero. End the game and stop all movement and machine work, but keep sending game updates.
Each welderWhen it has a board and material waiting, it needs 2,000 ms of work. Count down only while a nearby player is holding Solder. Pause when nobody is soldering.Put one material onto the board, remove it from the waiting material, reset the progress display, and make the board ready for testing.

3. Build and run

After you finish both missions, use three terminals: one for the display, one for the server, and one on each Raspberry Pi controller.

On the Ubuntu PC, build the server and start the display:

cd Lab4
make server

# Terminal 1: display
python3 ../Lab2/display.py --host 0.0.0.0 --port 17667

In a second Ubuntu terminal, start the game server. Use the number of players you have connected:

cd Lab4
./server --players 2 --input-port 17666 \
    --display-host 127.0.0.1 --display-port 17667

On each Raspberry Pi, load the Lab 4 driver once, then start one reader with a different player number:

cd ~/Lab4
sudo insmod overbug_buttons.ko debounce_ms=20
./reader --server <PC_IP> --player 1

# On the second Raspberry Pi:
./reader --server <PC_IP> --player 2

For three or four controllers, start more readers with player numbers 3 and 4, and use the same number after --players when starting the server.

  1. POSIX provides three interval-timer classes: real-time, virtual-time, and profiling timers. Which class does your server use?

  3. Compare a polling/sleep-loop server with an event-driven server that receives timer notifications. Which design is more appropriate for this game, and why?
  4. Analyse the two timer intervals used in this lab: 20 ms for button debounce and 50 ms for the game update. Do you consider these intervals appropriate?
  5. What other insights do you gain from this lab?
