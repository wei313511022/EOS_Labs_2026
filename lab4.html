1. Mission 1 — replace polling with IRQ, timer, and signal

The device name and text protocol do not change: /dev/overbug_buttons still returns OBTN XX\n. What changes is when the reader receives it. Remove the Lab 2 --hz option and all fixed-rate sleep logic. 

GPIO edge
  -> IRQ handler: wake the shared debounce timer
  -> timer callback: read six GPIO values and compare stable mask
  -> changed mask: store state, wake wait queue, signal SIGIO to reader
  -> reader SIGIO handler: wake the event loop
  -> reader read(): OBTN XX, then send one OBIN UDP packet

Driver contract

ComponentRequired behavior
GPIO setupRequest all six lines as inputs.
IRQRequest both rising and falling edges for every line. The handler must not read GPIO or notify userspace; it only restarts the shared debounce timer.
Debounce timerUse one timer_list and the read-only module parameter debounce_ms, defaulting to 20. The callback reads the complete mask and publishes it only if it differs from the last delivered mask.
Device readA normal read() blocks until a changed debounced mask is ready.
NotificationImplement a wait queue. Store the new state first, then call wake_up_interruptible() and kill_fasync(..., SIGIO, POLL_IN).
CleanupOn unload: prevent new opens, deregister the device, stop the timer permanently with timer_shutdown_sync(), free all IRQs/GPIO descriptors, and release the GPIO reference.

Reader contract

The reader is an event bridge, not a polling loop. SIGIO is a doorbell that means “the device may have a new state”; it does not contain the button mask. The reader obtains the authoritative state with read(), converts OBTN XX\n into the six-byte OBIN packet, and sends it to the game server. 

driver stores a changed stable mask
  -> driver sends SIGIO                 (doorbell only)
  -> reader handler sets a flag         (no read or UDP here)
  -> reader event loop reads OBTN XX    (real button data)
  -> reader sends OBIN to the server
  • Prepare safely: block SIGIO, then install sigaction(SIGIO). Its handler performs exactly one async-signal-safe action: set a volatile sig_atomic_t notification flag.
  • Subscribe to the device: open with O_NONBLOCK, use fcntl(fd, F_SETOWN, getpid()) to name this reader as the signal recipient, then enable O_ASYNC | O_NONBLOCK. O_ASYNC permits the driver's kill_fasync() call to deliver SIGIO; O_NONBLOCK makes an empty read() return EAGAIN rather than sleep.
  • Drain before sleeping: while SIGIO remains blocked, read and send every available OBTN state until read() returns EAGAIN. Signals can merge, so the device read—not the number of signals—is the source of truth.
  • Sleep without a race: after draining, call sigsuspend() with the old signal mask. It atomically unblocks SIGIO and sleeps. A button event therefore cannot arrive between “the queue is empty” and “the reader is asleep.”
  • Initialize server state: send a zero-mask packet once at startup to clear stale server movement. After that, send one OBIN packet for each state returned by the device.
block SIGIO; install handler; enable O_ASYNC
send OBIN(0) once
for (;;) {
    sigio_received = 0;
    while (read(device) returns OBTN XX)
        send OBIN(XX);
    /* read() returned EAGAIN; SIGIO is still blocked here */
    while (!sigio_received)
        sigsuspend(&old_mask);    /* unblock SIGIO and sleep atomically */
}

2. Mission 2 — timed multiplayer facilities

Copy your final Lab 3 server into Lab4/ first. Keep its multiplayer behavior and synchronization design, then add the timing rules below. Do not copy the repository's Lab 3 starter unless it is your completed submission. 

cd Lab4
cp /path/to/your/completed/Lab3/server.c server.c

Required interval timer

Configure setitimer(ITIMER_REAL, ...) for a repeating 50 ms interval and install a sigaction(SIGALRM) handler. The handler is only a wake-up mechanism: it may set one volatile sig_atomic_t flag or write one byte to a self-pipe, but it must not move players, update facilities, send snapshots, lock a mutex, or call printf().

SIGALRM is a standard signal, so several expirations may be merged into one delivered signal while the server is busy. In normal server code, use clock_gettime(CLOCK_MONOTONIC, ...) to calculate how many complete 50 ms intervals have elapsed since the last update. Apply every elapsed interval before publishing the next snapshot. Store countdowns as integer milliseconds, not floating-point wall-clock values. 

static volatile sig_atomic_t alarm_pending;

static void on_sigalrm(int signum)
{
    (void)signum;
    alarm_pending = 1;       /* async-signal-safe: wake normal server code */
}

/* Normal server code, never the signal handler: */
elapsed_ticks = monotonic_elapsed_ms_since_last_update() / 50;
for (i = 0; i < elapsed_ticks; ++i)
    advance_one_50_ms_game_tick();
alarm_pending = 0;

Continuous movement from event-driven input

A held direction button must keep a player moving, but the reader must not send repeated UDP packets while it is held. The driver generates an event only for the debounced press and release. On receiving OBIN, save that player's newest six-bit button mask. On every 50 ms timer expiration, move each player according to the saved mask. A release packet clears the direction and the next tick stops movement. 

reader:  press Right -> OBIN(Right) ---------------- release -> OBIN(no Right)
server:                save Right mask                         clear Right mask
50 ms tick:            move -> move -> move -> move -> stop

Therefore, replace any Lab 3 logic that moves a player once per received input packet. The UDP input handler only updates the stored mask; the timer owner applies movement and facility time. This keeps held movement smooth without turning the reader back into a polling sender.

Optional challenge — a select() event loop

This is optional. If you choose it, one process owns the authoritative game state and blocks on both the input UDP socket and the read end of a nonblocking self-pipe. The SIGALRM handler writes a byte to the pipe, so select() wakes for both UDP input and the interval timer without a check-then-sleep race. 

/* alarm handler: (void)write(tick_pipe[1], "T", 1); */
FD_ZERO(&read_fds);
FD_SET(input_sock, &read_fds);
FD_SET(tick_pipe[0], &read_fds);

if (select(max_fd + 1, &read_fds, NULL, NULL, NULL) >= 0) {
    if (FD_ISSET(input_sock, &read_fds))
        receive_all_valid_obin_packets(...);
    if (FD_ISSET(tick_pipe[0], &read_fds))
        drain_tick_pipe(...);
    advance_game_for_all_elapsed_50_ms_ticks(...);
    publish_current_obst_snapshot(...);
}

Students retaining their Lab 3 thread/process design may integrate SIGALRM within that design instead. In a threaded server, block SIGALRM before creating worker threads and designate one timer owner. Document who owns timer updates and how it prevents races with input and snapshots.

3. Verify before submitting

  1. Build make module reader. Confirm the reader help has no --hz option.
  2. Press and release one button. Confirm one stable state update per edge; deliberately bouncing a button must not create extra delivered states.
  3. Confirm the reader receives SIGIO, reads the device, and sends no periodic idle UDP packets.
  4. Unload the module both while idle and after the reader exits. There must be no active timer/IRQ callback or kernel warning.
  5. Run two to four players with the unchanged display. The HUD must count down from 03:00 to 00:00 and then display game over.
  6. Hold one movement button: the player must continue moving on successive 50 ms ticks, with only one press packet and one release packet from the reader.
  7. At a welder, verify that progress rises only during nearby held Solder, pauses after release/exit, and completes after 2 s of accumulated work.
  8. At a verifier, verify automatic 4 s progress, blocked early pickup, and correct pass/fail resolution.
  9. Operate separate welders/verifiers simultaneously. Each countdown must advance independently with no lost input or corrupted station state.
  10. If completing the optional challenge, show that select() wakes for the self-pipe timer event and UDP input without a sleep_ms()-based loop.

4. Submission and questions

  • Submit your modified Lab4/driver.c, reader.c, copied-and-modified server.c, Makefile, and a short test record.
  • Include a diagram of the IRQ → timer → SIGIO path and identify which operations are safe in IRQ, timer, and signal-handler contexts.
  • Explain why sigsuspend() is safer than checking a flag and then calling pause().
  • Explain how your timer ownership prevents two players or two facilities from racing on shared state.
  • Explain why SIGALRM is only a wake-up and how your monotonic-clock calculation recovers merged timer signals.
  • If you chose select(), explain how the self-pipe replaces the old polling/sleep loop and how you drain it safely.
