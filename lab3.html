Lab 3: Concurrent Multiplayer Server

Refactor the Lab 2 server into a Linux/POSIX threaded server that coordinates two to four Raspberry Pi players at the same time.

Before you begin

This lab builds directly on Lab 2. You should be comfortable with C, UDP sockets, and the Lab 2 input pipeline.

Linux/POSIX only Build and run the Lab 3 server on Ubuntu or Raspberry Pi OS. The threaded baseline uses pthread and POSIX semaphores; Windows support is not required.
  • Keep the six-byte OBIN input packet unchanged.
  • Reuse Lab2/reader.c on every Raspberry Pi and Lab2/display.py on the PC.
  • Use the copied Lab3/server.c as your single-player starting point.

System overview

Several Pi readers send input concurrently. Three server threads coordinate the shared game state and keep only the newest display snapshot.

Lab 3 runtime path

UDP OBIN :17666 mutex-protected
game state game thread latest snapshot
+ semaphore display thread
OBST :17667 Lab2/display.py

Students refactor Lab3/server.c. The existing reader and display programs remain the interfaces at either end of the server.

1. From Lab 2 to Lab 3

Lab 2 baseline Lab 3 target
One player, ID 1 Two to four active players, each with a distinct ID and spawn position
One loop receives input, updates the game, and sends snapshots Input, game, and display threads have separate responsibilities

2. Processes, threads, and shared state

Concept Meaning in this lab Synchronization
Process An independently running program with its own address space. Separate processes need IPC; shared data needs shared memory.
Thread An execution path inside one process. Lab 3 uses three threads. Threads share ordinary process memory, so access must be coordinated.
Mutex Exclusive access to the game state while input is written or the game advances. pthread_mutex_t
Semaphore An event notification that a new display snapshot is available. sem_t
Shared memory An optional later design if work moves into separate processes. POSIX shared memory plus process-shared synchronization
Required baseline Use one process with three threads. Do not add a shared-memory mapping for this lab. Shared memory matters only if you later split the input, game, or display work into separate processes.

A race condition occurs when the result depends on the timing of unsynchronized accesses. The mutex makes a critical section: only one thread can read or change the protected game state at a time.

3. Server requirements

  • Add --players N, accepting N = 2..4 and defaulting to 4.
  • Replace game.player with player storage and an active-player count. Initialize unique IDs and spawn positions.
  • Accept valid OBIN input for every active player ID. The packet stays exactly six bytes.
  • Update movement, collisions, station interactions, soldering, orders, and the OBST JSON snapshot for all active players.
  • Keep Lab2/reader.c, Lab2/display.py, UDP port 17666, and UDP port 17667 unchanged.
struct __attribute__((packed)) ob_input_packet {
    char magic[4];       /* "OBIN" */
    uint8_t player_id;   /* active player ID */
    uint8_t button_mask; /* 6-bit device state */
};

4. Thread design

Thread Responsibility Synchronization rule
Input Receive OBIN datagrams on port 17666 and update input masks for active IDs. Lock the game mutex only while reading or writing shared player state.
Game Advance movement, collisions, stations, orders, and score; then create a JSON snapshot. Lock the game mutex for the update and snapshot construction.
Display Wait for a fresh snapshot and send OBST JSON to port 17667. Wait on a semaphore, copy the snapshot under its mutex, then send after unlocking.

One latest-snapshot buffer

Keep a snapshot buffer, a snapshot_ready flag, a snapshot mutex, and a semaphore. When the game thread produces a frame, replace the buffer. Post the semaphore only when the flag changes from not-ready to ready. The display thread waits, copies the newest buffer, clears the flag, and sends it. A slow display therefore skips old frames instead of building a backlog.

Orderly shutdown

On SIGINT or SIGTERM, set a shared stopping flag, wake any thread blocked on the semaphore, join all threads, then close sockets and destroy mutexes and semaphores.

5. Refactor checklist

  1. Define MAX_PLAYERS 4; replace the single player with an array and active count.
  2. Parse --players; reject values outside 2–4; initialize IDs and non-overlapping spawns.
  3. Refactor input handling to select the matching active player and update only that player’s mask.
  4. Refactor game helpers to operate on each player. Resolve player-to-player collisions and contested stations safely.
  5. Serialize all active players in the JSON snapshot so the existing display can render them.
  6. Create the input, game, and display threads; add the mutex, latest-snapshot buffer, semaphore, and shutdown path.
struct shared_server {
    struct game game;
    pthread_mutex_t game_mutex;
    pthread_mutex_t snapshot_mutex;
    sem_t snapshot_available;
    bool snapshot_ready;
    bool stopping;
    char latest_snapshot[SNAPSHOT_BYTES];
};

/* Write these thread functions; keep each responsibility separate. */
static void *input_thread_main(void *arg);
static void *game_thread_main(void *arg);
static void *display_thread_main(void *arg);
Do not hold a mutex during network I/O. Copy the required shared data while locked, then unlock before a potentially slow recvfrom() or sendto() operation.

6. Build and run

After you implement the refactor, use three terminals on the PC or Ubuntu VM.

PC terminal 1: build the Lab 3 server

cd Lab3
make

PC terminal 2: start the provided display

python3 ../Lab2/display.py --host 0.0.0.0 --port 17667

PC terminal 3: start your multiplayer server

./server --players 2 --input-port 17666 --display-host 127.0.0.1 --display-port 17667

The supplied starter does not accept --players yet; this command should work after your refactor.

One Raspberry Pi per player

On each Pi, reuse the Lab 2 reader and give every controller a different ID:

cd Lab2
make reader
./reader --server <PC_IP> --player 1

# On the second Pi:
./reader --server <PC_IP> --player 2

For three or four players, start readers with --player 3 and --player 4 on additional Pi controllers, and start the server with the matching --players count.

7. QA

Check Expected observation
Run with 2, then 3, then 4 readers. Every active ID appears in the OBST snapshot and display.
Move players simultaneously into one another and toward a station. Collisions and contested interactions remain valid; no item or score corruption occurs.
Slow or pause the display temporarily. The display receives a recent state when it resumes; stale frames do not accumulate.
Stop the server with Ctrl+C. Threads exit, join, and release server resources without hanging.

Questions

  1. What race would occur if the input and game threads changed a player at the same time without a mutex?
  2. Why is a mutex for exclusive game-state access, while a semaphore is for notifying the display thread?
  3. Why does a single-process threaded design not need shared memory?
  4. When would POSIX shared memory and process-shared synchronization become necessary?
