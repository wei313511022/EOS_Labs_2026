Lab 1: System Call

Implement task_snapshot, a small kernel interface for observing how the calling program behaves while running.

Before you begin

Before starting this lab, make sure you have:

  • an Ubuntu build environment with the ARM64 cross-compiler installed;
  • a Raspberry Pi and boot media that you can use to test a rebuilt kernel; and

This lab assumes you understand:

  • User space vs kernel space: normal programs run in user space; privileged OS code runs in kernel space.
  • System call: a controlled entry point for a user program to request kernel service.
  • C struct: a fixed layout used to return several measurements together.
  • Pointer safety: the kernel must carefully copy data back to user memory.
  • Kernel build workflow: modify kernel code, rebuild, install, reboot, then test.
  • copy_to_user(): a kernel helper used to safely copy data back to a user program.

By the end, you will add a small system call to Linux, rebuild and boot the kernel, and call the new interface from a user-space C program.

Simple model for Lab 1 task_snapshot observes the program that calls it.

System call interface

In this lab, the system call receives one pointer from user space and fills that structure with runtime information. 

long task_snapshot(struct task_snapshot __user *buf);

Required behavior

  • The system call observes the calling program.
  • It writes one struct task_snapshot back to the user program.
  • Return 0 on success.
  • Return -EFAULT if the user pointer is invalid.

Student-facing structure

struct task_snapshot {
    uint32_t pid;
    uint32_t tgid;
    uint32_t cpu;
    uint64_t timestamp_ns;
    uint64_t runtime_ns;
    uint64_t user_cpu_ns;
    uint64_t system_cpu_ns;
    uint64_t voluntary_context_switches;
    uint64_t involuntary_context_switches;
    uint64_t minor_page_faults;
    uint64_t major_page_faults;
};
Field Meaning in Lab 1
pid ID of the running program/task.
tgid Group ID for the program. In simple Lab 1 programs, it is usually the same as pid.
cpu CPU core where the snapshot was taken.
timestamp_ns Kernel timestamp when the snapshot was taken.
runtime_ns Total time the program actually ran on a CPU.
user_cpu_ns CPU time spent running normal user program code.
system_cpu_ns CPU time spent inside the kernel on behalf of the program.
voluntary_context_switches How many times the program stopped because it waited, slept, or blocked.
involuntary_context_switches How many times the scheduler stopped the program to run something else.
minor_page_faults Memory faults handled without slow storage access.
major_page_faults Memory faults that required storage access and are usually much slower.

Step 1: Prepare the cross-compilation environment

Run these commands inside the Ubuntu 24.04 VM.

sudo apt update
sudo apt install -y git bc bison flex libssl-dev make libc6-dev libncurses5-dev \
    crossbuild-essential-arm64 dwarves
Why cross-compile? The VM is usually faster and more comfortable for building the kernel. The generated kernel image will run on the Raspberry Pi 4.

Step 2: Get the Raspberry Pi kernel source

Clone the Raspberry Pi Linux kernel. This handout uses the rpi-6.1.y branch so the later GPIO labs can demonstrate the legacy integer-based GPIO API. 

git clone --depth=1 --branch rpi-6.1.y https://github.com/raspberrypi/linux.git
cd linux

Step 3: Configure a 64-bit Raspberry Pi 4 kernel

Use the Raspberry Pi 4 default configuration and add a local version suffix so you can recognize your custom kernel after booting. 

make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- bcm2711_defconfig
scripts/config --set-str LOCALVERSION "-overbug-lab1"
scripts/config --disable LOCALVERSION_AUTO
make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- olddefconfig

Step 4: Add the UAPI header

Create include/uapi/linux/task_snapshot.h. User programs and kernel code must agree on this structure. 

#ifndef _UAPI_LINUX_TASK_SNAPSHOT_H
#define _UAPI_LINUX_TASK_SNAPSHOT_H

#include <linux/types.h>

struct task_snapshot {
    __u32 pid;
    __u32 tgid;
    __u32 cpu;
    __u64 timestamp_ns;
    __u64 runtime_ns;
    __u64 user_cpu_ns;
    __u64 system_cpu_ns;
    __u64 voluntary_context_switches;
    __u64 involuntary_context_switches;
    __u64 minor_page_faults;
    __u64 major_page_faults;
};

#endif

Step 5: Implement the system call

Create kernel/task_snapshot.c. The kernel uses current to refer to the program currently executing this system call. 

// SPDX-License-Identifier: GPL-2.0

#include <linux/errno.h>
#include <linux/ktime.h>
#include <linux/sched.h>
#include <linux/sched/cputime.h>
#include <linux/smp.h>
#include <linux/syscalls.h>
#include <linux/uaccess.h>

#include <uapi/linux/task_snapshot.h>

SYSCALL_DEFINE1(task_snapshot, struct task_snapshot __user *, user_snapshot)
{
    struct task_snapshot snapshot = { 0 };
    u64 user_time = 0;
    u64 system_time = 0;
    unsigned int cpu;

    if (!user_snapshot)
        return -EFAULT;

    snapshot.pid = (u32)task_pid_nr(current);
    snapshot.tgid = (u32)task_tgid_nr(current);

    cpu = get_cpu();
    snapshot.cpu = cpu;
    snapshot.timestamp_ns = ktime_get_ns();
    put_cpu();

    snapshot.runtime_ns = task_sched_runtime(current);

    task_cputime_adjusted(current, &user_time, &system_time);
    snapshot.user_cpu_ns = user_time;
    snapshot.system_cpu_ns = system_time;

    snapshot.voluntary_context_switches = (u64)current->nvcsw;
    snapshot.involuntary_context_switches = (u64)current->nivcsw;
    snapshot.minor_page_faults = (u64)current->min_flt;
    snapshot.major_page_faults = (u64)current->maj_flt;

    if (copy_to_user(user_snapshot, &snapshot, sizeof(snapshot)))
        return -EFAULT;

    return 0;
}

Step 6: Register the system call in the kernel build

Add the object file

Add this line to kernel/Makefile:

obj-y += task_snapshot.o

Add the prototype

Add this declaration to include/linux/syscalls.h:

struct task_snapshot;
asmlinkage long sys_task_snapshot(struct task_snapshot __user *user_snapshot);

Add the syscall table entry

Linux 6.1 registers generic ARM64 system calls in include/uapi/asm-generic/unistd.h. Immediately before the existing #undef __NR_syscalls line, add: 

#define __NR_task_snapshot 451
__SYSCALL(__NR_task_snapshot, sys_task_snapshot)

Then change the syscall count from 451 to 452:

#undef __NR_syscalls
#define __NR_syscalls 452
Check before editing. Confirm that the last assigned number is 450 and that __NR_syscalls is 451 before using syscall number 451. If they differ, ask the TA before choosing another number.

Step 7: Cross-compile the kernel

Build the kernel image, modules, and device trees.

make -j"$(nproc)" ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- Image modules dtbs

Record the kernel release string. You will need it when installing modules.

KREL=$(make -s ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- kernelrelease)
echo "$KREL"

Step 8: Install to the Raspberry Pi SD card

Insert the Raspberry Pi SD card into the VM or host. The examples below assume /dev/sdb1 is the FAT boot partition and /dev/sdb2 is the Linux root partition. Use lsblk -f first and replace those device names with the SD card shown on your machine.

Mount the boot and root partitions

lsblk -f
mkdir -p mnt/boot mnt/root
sudo mount /dev/sdb1 mnt/boot
sudo mount /dev/sdb2 mnt/root

BOOT="$PWD/mnt/boot"
ROOT="$PWD/mnt/root"
Check the device before mounting. Do not assume that the SD card is always /dev/sdb. Mounting the wrong partition can put the kernel files on the wrong disk.

Install kernel modules

STAGE="$PWD/stage-overbug-lab1"
mkdir -p "$STAGE"

make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- \
    INSTALL_MOD_PATH="$STAGE" INSTALL_MOD_STRIP=1 modules_install

sudo cp -a "$STAGE/lib/modules/$KREL" "$ROOT/lib/modules/"

Install the kernel image

sudo cp arch/arm64/boot/Image "$BOOT/kernel8-overbug-lab1.img"
sudo cp "$BOOT/config.txt" "$BOOT/config.txt.before-overbug-lab1"

Install Device Tree blobs and overlays

The Raspberry Pi firmware loads these files before starting the kernel, so install the Device Trees built from the same source tree as your custom kernel. 

sudo mkdir -p "$BOOT/overlays"
sudo cp arch/arm64/boot/dts/broadcom/*.dtb "$BOOT/"
sudo cp arch/arm64/boot/dts/overlays/*.dtb* "$BOOT/overlays/"
sudo cp arch/arm64/boot/dts/overlays/README "$BOOT/overlays/"
Device Tree files are shared. These commands replace the boot partition's existing base Device Trees and overlays; they do not create a second, per-kernel copy. Keep the SD card's original boot partition available if you later need to return to its original Raspberry Pi OS kernel.

Add these lines near the end of config.txt:

[all]
arm_64bit=1
kernel=kernel8-overbug-lab1.img

Safely flush writes and unmount the SD card:

sync
sudo umount "$BOOT"
sudo umount "$ROOT"
Keep a backup boot configuration. If the custom kernel does not boot, mount the SD card again and restore config.txt.before-overbug-lab1.

Step 9: Boot and verify the custom kernel

Boot the Raspberry Pi and connect through SSH.

ssh <pi-username>@<raspberry-pi-ip>
uname -r

The output should begin with 6.1 and include -overbug-lab1. If it does not, the Pi probably booted a different kernel image.

Step 10: Write the user-space test program (by yourself)

On the Raspberry Pi, create snapshot_test.c. This program calls syscall number 451 and prints the returned measurements.

Compile and run it:

gcc -std=c11 -O2 -Wall -Wextra snapshot_test.c -o snapshot_test
./snapshot_test

Required experiments

Test the following workloads.

Experiment What to do What to observe
NULL pointer test Call syscall(451, NULL). The call should fail and set errno to EFAULT.
CPU workload test Run a busy calculation between two snapshots. runtime_ns and user_cpu_ns increase.
Sleep/blocking test Sleep or wait between two snapshots. runtime_ns stays low; voluntary context switches may increase.
System-call-heavy test Call simple Linux system calls many times. system_cpu_ns increases more clearly.
Memory allocation test Allocate memory and touch new pages. minor_page_faults increases.

Expected results

  • NULL pointer: the system call fails safely with EFAULT.
  • CPU loop: high runtime_ns and high user_cpu_ns.
  • Sleep: low runtime_ns and more voluntary context switches.
  • System-call-heavy workload: increased system_cpu_ns.
  • Memory touch: increased minor page faults.

QA

  1. Why do we need a system call?
  2. What data did we measure? What does each field mean?
  3. Design your own test cases so that task_snapshot measures different results. Explain why the results are different.
  4. Use strace, ftrace, or another debugger/tracing tool to observe how your system call is passed through the system. Draw a flow chart and explain the path.
  5. What other insights do you gain from this lab?

Appendix: Restore the original kernel

Kernel development is easy to break. If your custom kernel cannot boot, or if you simply want to return to the original Raspberry Pi OS kernel, restore the backup config.txt created in Step 8.

Method A: The Raspberry Pi still boots

If the Pi can still boot and you can SSH into it, restore the boot configuration directly on the Pi. 

ssh <pi-username>@<raspberry-pi-ip>

if [ -d /boot/firmware ]; then
    BOOT_ON_PI=/boot/firmware
else
    BOOT_ON_PI=/boot
fi

sudo cp "$BOOT_ON_PI/config.txt.before-overbug-lab1" "$BOOT_ON_PI/config.txt"
sudo reboot

After rebooting, verify that the custom kernel suffix is gone:

uname -a

Method B: The Raspberry Pi cannot boot

If the Pi cannot boot, power it off, remove the SD card, and mount the SD card on the Ubuntu VM or host computer. 

lsblk -f

Find the boot partition. It is usually mounted as bootfs.

BOOT=/media/$USER/bootfs

sudo cp "$BOOT/config.txt.before-overbug-lab1" "$BOOT/config.txt"
sync
sudo umount "$BOOT"

Put the SD card back into the Raspberry Pi and boot again. The Pi should now use the original kernel configuration.

If the backup file is missing

If config.txt.before-overbug-lab1 does not exist, open config.txt and remove or comment out the custom kernel line. 

sudo nano "$BOOT/config.txt"

Remove or comment out this line:

kernel=kernel8-overbug-lab1.img
Do not delete the custom kernel image first. First restore config.txt and confirm the Pi can boot. After the original kernel boots successfully, you may clean up old custom kernel images and module folders later.
